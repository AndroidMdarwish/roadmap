		<title>Codespaces support for JetBrains Rider (Beta)</title>
		<link>https://github.blog/changelog/2023-02-24-codespaces-support-for-jetbrains-rider-beta</link>
		
		<dc:creator><![CDATA[Kevin Duck]]></dc:creator>
		<pubDate>Fri, 24 Feb 2023 22:16:33 +0000</pubDate>
				<guid isPermaLink="false">https://github.blog/changelog/2023-02-24-codespaces-support-for-jetbrains-rider-beta</guid>

					<description><![CDATA[Codespaces support for JetBrains Rider (Beta)]]></description>
										<content:encoded><![CDATA[<p>The <a href="https://plugins.jetbrains.com/plugin/20060-github-codespaces">GitHub Codespaces plugin</a> for the <a href="https://www.jetbrains.com/remote-development/gateway/">JetBrains Gateway</a> now supports Rider as a remote IDE. .NET developers can now leverage the standardization and power of GitHub Codespaces with JetBrains Rider&#039;s singular code indexing, navigation, and debugging capabilities.</p>
<p><img decoding="async" src="https://i0.wp.com/user-images.githubusercontent.com/4679612/221053533-adbc24fc-9adf-49bc-81fa-82afb159924f.png?ssl=1" alt="JetBrains Rider in Gateway" data-recalc-dims="1"></p>
<p>GitHub Codespaces support for Rider enables multiple solution file scenarios. If there is only one solution file in a given codespace, the GitHub Codespaces plugin will automatically select that solution file. If there are multiple, the plugin will prompt the user to select which solution file they intend to use to open their project. Repositories without solution files are still compatible with Rider, however <a href="https://www.jetbrains.com/help/rider/Creating_and_Opening_Projects_and_Solutions.html#different-ways-to-open-solutions">some features will be limited</a> when no solution file is selected.</p>
<p><img decoding="async" src="https://i0.wp.com/user-images.githubusercontent.com/4679612/221269506-563015a7-b1b5-4ca9-bb81-c6e58f000425.png?ssl=1" alt="Rider solution file picker" data-recalc-dims="1"></p>
<p>To get started with Rider, follow <a href="https://docs.github.com/en/codespaces/developing-in-codespaces/using-github-codespaces-in-your-jetbrains-ide">the documentation for installing GitHub Codespaces</a> into the JetBrains Gateway. Once installed, users can connect to any of their existing codespaces with Rider as their selected IDE.</p>
<p>We are extremely excited to deliver our <a href="https://github.com/github/codespaces-jetbrains-feedback/issues/5">top requested feature</a> since the beta announcement of JetBrains support in GitHub Codespaces.</p>
<p><strong>Additional Resources:</strong></p>
<ul>
<li><a href="https://docs.github.com/en/codespaces/developing-in-codespaces/using-github-codespaces-in-your-jetbrains-ide">GitHub&#039;s documentation</a> for using Codespaces with JetBrains IDEs.</li>
<li><a href="https://www.jetbrains.com/rider/documentation/">JetBrains Rider documentation</a>.</li>
<li><a href="https://github.com/github/codespaces-jetbrains-feedback">Feedback repository</a>.</li>
<li><a href="https://github.com/dotnet/samples">Example dotnet repository</a>.</li>
</ul>

										<content:encoded><![CDATA[<p>Reading and understanding code is an absolutely critical task for software developers. Research suggests developers spend far more time reading code than writing it. Reviewing a pull request, planning a new feature, researching a system‚Äôs architecture, or determining how to fix a bug are all activities that rely on finding critical information scattered across the codebase.</p>
<p>That‚Äôs why we‚Äôve built the new code search and code view‚Äîto help developers search, navigate, and understand their code, their team‚Äôs code, and the world‚Äôs open source code.</p>

<video class="wp-video-shortcode js-viewport-aware-video events-none" id="video-70282-1" width="1024" height="360" poster="https://github.blog/wp-content/uploads/2023/02/Screenshot-2023-02-23-at-9.00.57-PM.png" loop="1" autoplay="1" preload="metadata" playsinline="1" muted="1"><source type="video/mp4" src="https://github.blog/wp-content/uploads/2023/02/code_search_code_view_small.mp4?_=1" /><a href="https://github.blog/wp-content/uploads/2023/02/code_search_code_view_small.mp4">https://github.blog/wp-content/uploads/2023/02/code_search_code_view_small.mp4</a></video></div>

<p><img decoding="async" src="https://i0.wp.com/user-images.githubusercontent.com/102542428/221022796-14354d34-95c4-4dbc-a9fa-7af8f7b2e763.png?ssl=1" alt="mockup screenshot of new code view and code search features" data-recalc-dims="1" /></p>

<p>After configuring and enabling the initial auto-add workflow, open the context menu in the workflow list and select <code>Duplicate workflow</code> to create a new auto-add workflow.</p>

					<description><![CDATA[Hardware accelerated Android virtualization on Actions Windows and Linux larger hosted runners]]></description>
										<content:encoded><![CDATA[<p>Starting on February 23, 2023, Actions users of GitHub-hosted <a href="https://docs.github.com/en/actions/using-github-hosted-runners/using-larger-runners">larger Linux runners</a> will be able to make use of hardware acceleration for Android testing. Testing on a 4-core machine with hardware acceleration is around 2-3 times faster than not using hardware acceleration and around 2 times faster than using MacOS. </p>
<p>To make use of this on Linux, Actions users will need to add the runner user to the KVM user group</p>
<pre><code>      - name: Enable KVM group perms
        run: |
            echo &#039;KERNEL==&quot;kvm&quot;, GROUP=&quot;kvm&quot;, MODE=&quot;0666&quot;, OPTIONS+=&quot;static_node=kvm&quot;&#039; | sudo tee /etc/udev/rules.d/99-kvm4all.rules
            sudo udevadm control --reload-rules
            sudo udevadm trigger --name-match=kvm</code></pre>
<p>(Thank you <a href="https://github.com/gsauthof">gsauthof</a> for the feedback on this!)</p>
<p>You will then be able to make use of hardware acceleration when making use of Android emulator actions such as <a href="http://github.com/reactivecircus/android-emulator-runner">reactivecircus/android-emulator-runner</a>.</p>

					<description><![CDATA[Preferred 2FA methods and settings improvements]]></description>
										<content:encoded><![CDATA[<p>The <code>Primary</code> field on two-factor authentication methods has been removed, and replaced with a <code>Preferred</code> option. This new option sets your preferred 2FA method for account login and use of the sudo prompt. You can choose between TOTP, SMS, security keys, or GitHub Mobile as your preferred 2FA method.</p>
<p>Additionally, you can now update your 2FA methods inline at <a href="https://github.com/settings/security">https://github.com/settings/security</a>, rather than going through the initial 2FA setup flow again.</p>
<p><img decoding="async" src="https://i0.wp.com/user-images.githubusercontent.com/1666363/219820201-6fe7f80a-cb6b-4bb9-bd56-984ed917e6a5.png?ssl=1" alt="image" data-recalc-dims="1"></p>
<p>With this change, device-specific preferences for 2FA have been removed &#8211; each login will always default to your preferred method. If you previously set a default on one of your devices, your most recent choice has been copied to your account-wide preference. Otherwise, no preference will be set, and GitHub will select from your available second factors in this order: security keys, GitHub Mobile, TOTP, and then SMS.</p>
<p>To learn more, see <a href="https://docs.github.com/en/authentication/securing-your-account-with-two-factor-authentication-2fa/changing-your-preferred-two-factor-authentication-method">&quot;Changing your preferred two-factor authentication method&quot;</a> and <a href="https://docs.github.com/en/authentication/securing-your-account-with-two-factor-authentication-2fa/configuring-two-factor-authentication">&quot;Configuring two-factor authentication&quot;</a>.</p>

def run_during(duration, func):
    _t = time.time
    n = 0
    start = os.times()
    start_timestamp = _t()
    real_start = start[4] or start_timestamp
    while True:
        func()
        n += 1
        if _t() - start_timestamp > duration:
            break
    end = os.times()
    real = (end[4] if start[4] else time.time()) - real_start
    return n, real, sum(end[0:2]) - sum(start[0:2])

def prepare_files():
    print("Preparing files...")
    # Binary files
    for name, size in get_binary_files():
        if os.path.isfile(name) and os.path.getsize(name) == size:
            continue
        with open(name, "wb") as f:
            f.write(os.urandom(size))
    # Text files
    chunk = []
    with text_open(__file__, "r", encoding='utf8') as f:
        for line in f:
            if line.startswith("# <iobench text chunk marker>"):
                break
        else:
            raise RuntimeError(
                "Couldn't find chunk marker in %s !" % __file__)
        if NEWLINES == "all":
            it = itertools.cycle(["\n", "\r", "\r\n"])
        else:
            it = itertools.repeat(
                {"cr": "\r", "lf": "\n", "crlf": "\r\n"}[NEWLINES])
        chunk = "".join(line.replace("\n", next(it)) for line in f)
        if isinstance(chunk, bytes):
            chunk = chunk.decode('utf8')
        chunk = chunk.encode(TEXT_ENCODING)
    for name, size in get_text_files():
        if os.path.isfile(name) and os.path.getsize(name) == size:
            continue
        head = chunk * (size // len(chunk))
        tail = chunk[:size % len(chunk)]
        # Adjust tail to end on a character boundary
        while True:
            try:
                tail.decode(TEXT_ENCODING)
                break
            except UnicodeDecodeError:
                tail = tail[:-1]
        with open(name, "wb") as f:
            f.write(head)
            f.write(tail)
